# STM → OpenMetadata Enricher

Runs the full "catalog + enrich" flow for a set of dbt-produced dim/fact tables, using the corresponding STM markdown files as the authoritative source for descriptions, classification tags, and glossary terms.

This skill orchestrates existing primitives — `openmetadata-sync` for ingestion, `openmetadata-glossary-tagger` for term assignment — and adds the STM-driven enrichment loop and concurrency-safe subagent pattern.

## When to use

- After a dbt Cloud or local dbt run produces new tables in a Snowflake schema (e.g. `DBT_PROD_ENRICHED`).
- STMs exist in a known folder (default `stm/output/*.md`) and were generated by `stm-from-data-model` / `stm-catalogue-enricher`.
- You want the OpenMetadata catalog to reflect those STMs: table purpose as description, Section 5 tags, Section 6 glossary terms, Section 7 column descriptions.

## Prerequisites

- OpenMetadata MCP server registered as `user-openmetadata`.
- `.env` contains `OPENMETADATA_BASE_URL`, `OPENMETADATA_EMAIL`, `OPENMETADATA_PASSWORD`.
- A Snowflake database service already exists in OpenMetadata with working credentials. If not, run `openmetadata-sync` first to create it.
- STM files follow the layout produced by `stm-from-data-model` (Sections 2, 5, 6, 7 Final present).

## Workflow

Copy this checklist and track progress:

```
- [ ] Step 1: Preflight — inspect STMs and decide target schemas
- [ ] Step 2: Widen ingestion pipeline filter to include dbt schemas
- [ ] Step 3: Trigger ingestion and wait for success
- [ ] Step 4: Verify all STM target tables exist in OpenMetadata
- [ ] Step 5: Fast-path apply STMs via apply_stm_to_openmetadata.py
- [ ] Step 6: Subagent fallback ONLY for items the script couldn't handle
- [ ] Step 7: Verify enrichment with spot-check script
```

The fast path (Step 5) is purely programmatic and finishes in ~10–15s for
10 tables. Subagents (Step 6) are reserved for ambiguous cases the script
flags (STM column not found in OM, unknown tag/glossary FQN, failed PATCH).

### Step 1 — Preflight

1. List STM files (default glob: `stm/output/*.md`, skip `README.md`).
2. For each STM, extract from Section 4 the target schema name (e.g. `DBT_PROD_ENRICHED`) and target table (e.g. `DimCustomer`).
3. Ask the user if the dbt run targeted **prod** (`DBT_PROD*`) or **dev** (`DBT_DEV*`). The dev profile writes to `DBT_DEV` / `DBT_DEV_ENRICHED`; dbt Cloud prod writes to `DBT_PROD` / `DBT_PROD_ENRICHED`. Do not guess.
4. Compute the schema set to ingest: `{<layer views schema>, <enriched tables schema>}` e.g. `{DBT_PROD, DBT_PROD_ENRICHED}`, plus any existing included schemas the pipeline already has (to preserve prior catalog coverage).

### Step 2 — Widen ingestion pipeline filter

The MCP `update_metadata_ingestion_pipeline` tool frequently returns opaque 400 errors. **Use the REST PATCH helper instead**:

```bash
python scripts/patch_pipeline_filter.py \
  --pipeline-id <uuid> \
  --include-schemas BRONZE_ERP__DBO,DBT_PROD,DBT_PROD_ENRICHED \
  --include-databases DRIP_DATA_INTELLIGENCE
```

Pipeline id can be discovered with `list_ingestion_pipelines` (MCP). The script handles login, base64-password quirk, and JSON-patch payload.

### Step 3 — Trigger ingestion

1. Call `run_ingestion_pipeline` with `pipeline_name=<service>.<pipeline>` (FQN, not short name — short name returns 404).
2. If the first trigger response says `Workflow [...] has been created` but no new run appears, call the REST trigger endpoint once more (the MCP's first call sometimes only deploys the DAG, the second one actually queues a run):

```bash
python scripts/wait_for_ingestion.py --pipeline-id <uuid> --timeout 300
```

The script polls `?fields=pipelineStatuses` and returns when `pipelineState=success` or the timeout is hit.

### Step 4 — Verify tables exist

For every STM target table, call `list_tables` on the enriched schema FQN and assert each `<TABLE>` (uppercased) is present. If any are missing, stop and report — do not proceed to enrichment with a partial catalog.

### Step 5 — Fast-path apply (programmatic)

Run `apply_stm_to_openmetadata.py` over the STM folder. It parses Sections
2/4/5/6/7, fetches each OM table, and issues ONE JSON-patch per table that
updates description + column descriptions and merges new tags/glossary terms
(without removing existing ones). For 10 STMs this takes ~10–15 seconds.

```bash
python scripts/apply_stm_to_openmetadata.py \
  --stm-dir stm/output \
  --service snowflake_fivetran \
  --database DRIP_DATA_INTELLIGENCE \
  --schema DBT_PROD_ENRICHED
```

Or for a single STM with an explicit FQN:

```bash
python scripts/apply_stm_to_openmetadata.py \
  --stm stm/output/05-DimCustomer-stm.md \
  --table-fqn snowflake_fivetran.DRIP_DATA_INTELLIGENCE.DBT_PROD_ENRICHED.DIMCUSTOMER
```

The script prints a per-STM summary (counts of desc/tags/glossary applied,
PLUS any STM columns that could not be matched to an OM column and any REST
errors). Capture this output — the "missing" / "error" rows drive Step 6.

Handled deterministically by the script:

- Section 2 blockquote → table description (idempotent replace)
- Section 4 target schema+table → target FQN composition
- Section 5 Scope=Table rows → table-level classification tags (merged)
- Section 5 Scope=Column rows → column-level classification tags (merged)
- Section 6 Scope=Table rows → table-level glossary terms (merged, stored
  under `/tags` with `source=Glossary`)
- Section 6 Scope=Column rows → column-level glossary terms (merged)
- Section 7 Final "Description" → column descriptions (idempotent replace)
- PascalCase STM column → UPPERCASE OM column via normalized match
- Type-2 STMs with both "Data Condition N" and "Final" tables: the script
  picks the "Final" table by scanning for the one whose header contains both
  `target column` and `description`.

### Step 6 — Subagent fallback (only for what the script couldn't handle)

Look at the Step 5 report. Only launch a subagent if one of these is true:

- `missing` column count > 0 for a table (STM column name that did not match
  any OM column — e.g. `HashPK` renamed at ingestion, or typo in STM)
- a row is marked `[FAIL]` with a REST error (unknown tag FQN, glossary term
  not yet published, 4xx on PATCH)

Prompt template per fallback subagent:

```
The programmatic applier reported issues on TABLE {TABLE_FQN}.
Unmatched STM columns: {LIST}
REST errors: {LIST}
STM FILE: {STM_PATH}

Use CallMcpTool on server `user-openmetadata`:
- get_table to confirm actual OM column names
- assign_tags_to_column / assign_glossary_term_to_column / update_column_description
- If the tag or glossary FQN truly does not exist in OM, STOP and report it
  so governance-vocab-generator / openmetadata-vocab-publisher can add it.

Make MCP calls sequentially. Report what you fixed vs what you escalated.
```

Do not launch subagents "just in case" — the fast path is authoritative for
every row it processed. Launching ≥4 subagents in parallel has empirically
overwhelmed the OM MCP; keep parallelism ≤ 4 even in fallback mode.

### Step 7 — Verify

Run the verification helper on the enriched schema:

```bash
python scripts/verify_enrichment.py \
  --schema-fqn snowflake_fivetran.DRIP_DATA_INTELLIGENCE.DBT_PROD_ENRICHED \
  --tables DIMCUSTOMER,DIMDATE,DIMPRODUCT,...
```

For each table, the script asserts:
- non-empty table description
- at least one table-level tag
- at least one column description

Reports a pass/fail table and exits non-zero if any table fails.

## STM parsing rules

| STM location | Maps to | Notes |
|---|---|---|
| Section 2 "Purpose / Use Case" prose | table description | Single paragraph; no added prefix. |
| Section 4 target table + schema | target FQN | `<service>.<database>.<target schema uppercased>.<table uppercased>` |
| Section 5 `Scope=Table` | `assign_tags_to_table` | Full FQN e.g. `Architecture.Enriched`. |
| Section 5 `Scope=Column` | `assign_tags_to_column` | Column name must be UPPERCASED from STM's PascalCase. |
| Section 6 `Scope=Table` | `assign_glossary_term_to_table` | Full FQN e.g. `RetailDomainGlossary.Customer`. |
| Section 6 `Scope=Column` | `assign_glossary_term_to_column` | Column name UPPERCASED. |
| Section 7 Final "Description" col | `update_column_description` | Skip empty cells. |

## Utility scripts

- `scripts/patch_pipeline_filter.py` — REST PATCH the ingestion pipeline's `schemaFilterPattern` and `databaseFilterPattern`. Workaround for the flaky MCP `update_metadata_ingestion_pipeline` tool.
- `scripts/wait_for_ingestion.py` — Poll a pipeline's latest run until it reaches `success` / `failed` or a timeout elapses.
- `scripts/apply_stm_to_openmetadata.py` — Parse STMs and issue a single JSON-patch per table covering description + merged tags/glossary + column descriptions. Idempotent. Runs in ~1s per table. Primary enrichment path — subagents are only a fallback.
- `scripts/verify_enrichment.py` — Spot-check every enriched table and report missing descriptions/tags.

All scripts load credentials from `.env` (`OPENMETADATA_BASE_URL`, `OPENMETADATA_EMAIL`, `OPENMETADATA_PASSWORD`).

## Guardrails

- Do not drop existing schemas from the pipeline filter — always union with current includes.
- Do not pick `DBT_DEV*` schemas when the user's dbt run was in prod (and vice versa). Ask if unsure.
- Do not launch more than 4 enrichment subagents in parallel.
- Do not create classifications or glossary terms that are missing in OpenMetadata — surface them for review instead. Creation is the job of `governance-vocab-generator` / `openmetadata-vocab-publisher`.
- Do not remove prior tags or descriptions. All operations are additive/overwrite-on-same-field.
